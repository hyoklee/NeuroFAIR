## [2026-06-02] perf | Case 7 with the HDF5 VOL CTE adapter — no measurable number (ares)

User asked to run MiV case 7 through the **HDF5 VOL** CTE adapter (not the POSIX LD_PRELOAD adapter all prior case-7 studies used) and measure whether clio-core beats the native-HDF5 baseline. Investigated the VOL connector source `~/core.iowarp/context-transfer-engine/adapter/hdf5_vol/iowarp_vol.{cc,h}` (connector `iowarp`, value 600). **No performance number is obtainable**, for a stronger reason than the POSIX adapter: (A) it is unbuilt — `CLIO_CTE_ENABLE_HDF5_VOL:BOOL=OFF` in `~/core.iowarp/build/CMakeCache.txt`, and its `CMakeLists.txt` `return()`s unless HDF5≥2.0 while ares has only 1.14.x; the guard is over-conservative — 1.14.6 already has `H5VL_VERSION 3` and the callback signatures match the 1.14.6 `H5VLconnector.h` exactly (a `-fsyntax-only` compile showed zero HDF5-version errors), so a one-line guard relaxation would build it. (B) No `H5PLget_plugin_*` export → can't be loaded via `HDF5_VOL_CONNECTOR` env var → requires modifying MiV/neuroh5 to call `H5VL_iowarp_register`+`H5Pset_vol`. (C) **Decisive:** `iowarp_dataset_open` always sets `dset->file`, so `iowarp_dataset_read` serves data **only from CTE blobs** (`AsyncGetBlob` of `<path>/chunk_i`) with **no native-file fallback** and ignores `file_space_id` hyperslab selections. It's a write-through-cache prototype: data must be written through the connector first. Case 7 reads pre-existing native `.h5` files (MiV_Cells 19 GB, MiV_Connections 7 GB) → blob miss → zero-filled buffers → neuroh5 crashes at init on corrupt structure. Created `wiki/miv_iowarp_ares_case7_vol.md`; updated `index.md`. Verdict: the VOL adapter is a write-through cache, case 7 is a read-existing-files workload — they don't meet until the read path gains native-fallback + hyperslab support upstream.

## [2026-05-27] perf | Write-bound MiV benchmark designed to favor CTE — still ~11% slower on small circuit (ares)

Designed and implemented a write-bound MiV workload meant to let IOWarp CTE beat baseline: `run-network` with a dense `IO_heavy` recording profile (v from every section of every cell, dt=0.025) + `--checkpoint-interval=10` streams periodic HDF5 writes; baseline writes to NFS `/mnt/common`, IOWarp redirects to the node-local CTE tier. Measured the ares I/O hierarchy (O_DIRECT): NFS 590 MB/s read / 215 write, local NVMe 1.5 GB/s / 1.0, RAM ~10 GB/s, page cache only 46 GB/node — so ares *does* have a slow shared baseline (NFS) + small cache. **Result: CTE still ~11% slower** (baseline reached sim t=335 ms vs CTE 296–299 ms at equal ~1140 s wall; RAM≈NVMe). Reason: dense recording inflates NEURON *compute* (which the POSIX adapter slows via syscall interception) while total write volume stays small (tens of MB), so the write-bandwidth gain is negligible — same compute-bound story as cases 4/6/7. Documented the **recipe** for when CTE would win (I/O ≫ compute, re-read/write, working set > 46 GB or multi-node) and why the small circuit can't reach it (scaling I/O scales compute; full-scale data not staged on ares; multi-node CTE deadlocks). Created `wiki/miv_iowarp_ares_caseio.md`; artifacts `~/bin/run_caseio.sbatch`, `chimaera_caseio{,_nvme}.yaml`, `IO_heavy` profile; data `4-opsin/results/caseio_writebound.csv`. Jobs 20418–20420.
